개발 문화: agile: waterfall model: 가장 익숙한 소프트웨어 개발 기법 고전적인 소프트웨어 생명 주기 병행 수행디지 않고 순차적으로 진행 장점 단순한 모델이라 이해가 쉽다 단꼐별로 정형화된 접근이 가능해 문서화가 가능하다 프로젝트 진행 상황을 한눈에 명확하게 파악 가능하다 단점 변경을 수행하기 어렵다 시스템의 동작을 후반에 가야지만 확인이 가능하다 대형프로젝트에 적용하는 것이 부적합하고, 일저이 지연도리 가능성이 크다. aigle(사상): 에자일이라는 사상을 가지고 스크럼이라는 방법을 사용하는 형태 2주에서 4주정도로 짧은 주기를 잡고 설계, 개발, 테스트 배포 과정을 반복 요구상을 작은 단위로 쪼개 그에 대한 솔루션을 만들고, 빠르게 보여줌으로써 요구 사항에 대한 검증을 진행 scrum(방법론) 2주에서 4주동안 설계, 개발, 테스트 배포를 하는 방법 kanban: to do, doing, done으로 해야할것, 하고 있는 것, 끝낸 것을 나눠서 업무 흐름을 측정과 관리 github에서도 kanban보드 시스템 제공 git: 소스 코드를 효율적으로 관리하기 위해 만들어진 "분산형 버전 관리 시스템" branch: 독립적으로 어떤 작업을 하기 위해 필요한 개념 git branch: 내 로컬 레파지토리에 몇개의 브랜치가 있는지 확인 하는 방법 각각의 가지에서 add,commit,push가 가능 git branch "브랜치 명": 새로운 브랜치 만들기 git checkout "브랜치 명": 다른 브랜치로 이동하기(옮기기) 메인 이외의 다른 브랜치에서 push할때: git push origing 브랜치명 -내 로컬에서 브랜치 지우는 법 git branch -d 브랜치 명: 최신에 바뀐것이 메인에 반영되지 않았다고 알림이 온다. git branch -D 브랜치 명 반영하고 싶다면, d를 대문자D로 바꾸면 반영 -github 레파지토리에서 브랜치 지우는 법 git push origin --delete 브랜치 명 git checkout -b "브랜치 명" : 브랜치를 만들고 그 브랜치로 이동하겠다. merge: branch에 branch를 합치는 것 1. git checkout 배이스 브랜치 입력 : 합쳐지는 베이스 브랜치로 이동 git merge 머지 브랜치 입력: 베이스 브랜치에 합쳐질 머지 브랜치로 이동 i를 누르면 커밋 메세지 수정 가능 wq! : 메세지를 수정했을때 머지 q! : 메세지를 수정하지 않았을 때 머지 merge의 case: 1. a브랜치와 b브랜치에서 서로 다른 파일을 수정했을 때 =>위 과정을 통해 자동으로 머지 ex) a브랜치 1파일, b브랜치 2파일 수정 2. a브랜치와 b브랜치에서 서로 같은 파일의 다른 부분을 수정했을 때 => 위과정을 통해 자동으로 머지 ex) a브랜치 1파일의 title태그와, b브랜치 1파일의 head태그 수정 3. a브랜치와 b브랜치에서 서로 같은 파일의 같은 부분 수정했을 때 => conflict(충돌)이 발생함으로 머지할 것을 ex) a브랜치 1파일의 title태그와, b브랜치 1파일의 title태그 수정 vs코드 수정해야되는 부분 위에 선택사항이 뜬다 선택사항 1. Accept Current Change: 머지하려는 베이스 브랜치의 내용으로 변경 2. Accept Incoming Change: 머지하는 머지 브랜치의 내용으로 변경 3. Accept Both Change: 베이스 브랜치 내용과 머지 브랜치 내용 모두 적용(변경) 4. Compare Change: 변경사항 비교 선택사항 선택 후 파일 저장 -> add -> commit -> push 과정을 거쳐야 한다. Branch 종류 main(master): 제품으로 출시될 수 있는 브랜치 배포 이력을 관리하기 위해 사용 배포 가능한 상태만을 관리하는 브랜치 develop: 다음 출시 버전을 개발하는 브랜치 기능 개발을 위한 브랜치들을 병합하기 위해 사용 평소 개발을 진행하는 브랜치 개발 단계에서 발생한 버그를 수정 feature: 기능 개발을 진행하는 브랜치 새로운 기능 개발 및 버그 수정을 할 때마다 'develop'애서 분기 공유할 필요가 없어 로컬에서 진행후 develop에 merge해 공유 release: 출시 버전을 준비하는 브랜치 배포를 위한 전용 브랜치 이름: release-0.0 hotfix: 출시 버전에서 발생한 버그 수정 브랜치 배포한 버전에 긴급하게 수정해야 할 필요가 있는 경우 사용 master에서 분기 이름 hotfix-0.0.0 git git commit -am "커밋 메세지": git add + git commit 단점: 특정한 것만 올릴 수 없고 모두다 반영 ex) git add readme.md => readme 파일만 올릴 수 있음 git commit -ad => 파일 모두가 올라감 git add 취소: vs코드 source control(왼쪽 위에서 3번째) 가서 -버튼 누르면 되돌리기 가능 git commit 취소: git rest HEAD^: 가장 최근에 한 커밋 하나가 취소 git log: id 찾기 git reset --hard 찾은id: id 시점까지의 모든 것이 리셋(잘 사용하지 않음) pull request: 내가 수정한 코드가 있으니 내 branch를 가져가 검토 후 병합(merge)해주세요 base branch: 내가 지금 올릴 곳 .gitignore: git에서 제외할 파일 목록을 지정하는 파일 vs코드에서 .gitignore파일을 만들고 그 파일 내용에 무시 할 파일 이름을 적으면 그 파일을 무시하고 관리 라이브러리, 비밀번호 등을 보통 넣음 그러나, 깃에 한번도 올라가지 않은 파일만 무시할 수 있음 .txt: 확장자가 txt로 끝나는 파일 모두 무시하고 !test.txt: 해당 파일은 무시하지 않겠다. test/: test 폴더 내부의 모든 파일을 무시(b.exe와 a.exe 모두 무시) /tset: (현재폴더) 내에 존재하는 폴더 내부의 모든 파일 무시 (b.exe무시)
